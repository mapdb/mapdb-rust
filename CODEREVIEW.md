# Code Review: mapdb-rust

## Overview

Zero-dependency Rust primitive collections library achieving true type specialization. **492 source files**, **89,107 lines**, **3,707 test functions**. Implements hash maps, hash sets, array lists, bags, stacks, tree maps/sets, multimaps, bimaps, tuples, streams, and immutable variants for all 8 primitive types (i8, i16, i32, i64, f32, f64, bool, char) and their combinations.

**Rating: 9/10** — Exemplary zero-dependency Rust library with professional-grade hash table implementation and comprehensive test coverage.

---

## 1. Architecture & Design

### Module Organization

| Module | Purpose |
|--------|---------|
| `hashmap/` | Open-addressing hash tables (64 type combinations) |
| `hashset/` | Hash-based sets (8 primitive types) |
| `arraylist/` | Dynamic array-backed lists |
| `bag/` | Multisets (occurrence counting) |
| `stack/` | LIFO stacks |
| `treemap/` / `treeset/` | Sorted collections (delegates to `BTreeMap`/`BTreeSet`) |
| `multimap/` | One-to-many maps |
| `tuple/` | Immutable pairs (64 combinations) |
| `stream/` | Lazy iteration pipelines |
| `traits/` | Read-only trait boundaries |
| `immutable/` | `Arc<[T]>`-backed snapshot wrappers |

### Core Design Decisions

**Type Specialization Strategy**: Each primitive type gets its own generated struct (`I32ArrayList`, `I32HashSet`, `I32I64HashMap`). This eliminates boxing and achieves true zero-cost abstractions — compiled output is nearly identical to hand-written `Vec` operations.

**Generic Trait Infrastructure**: `OpenHashMap<K,V>` and `OpenHashSet<K>` in `hash_table.rs` (679 lines) serve as the foundation. All 264 specialized types build on these generics.

**Ownership Model**:
- Arrays/HashSets own data via `Vec<T>` — simple lifetime semantics
- `Arc<[T]>` for immutable variants — cheap O(1) clones, thread-safe sharing
- BiMaps maintain two independent `OpenHashMap` instances
- No lifetime parameters in generated code — keeps output simple and maintainable

**Zero `unsafe` Code**: Verified — 0 instances across all 492 files. All bounds checking relies on Rust's standard library.

## 3. Performance

### Hash Table Implementation (`hash_table.rs`)

**Interleaved Cache-Friendly Entry Layout** (lines 84–97):
```rust
struct MapEntry<K, V> {
    occupied: bool,
    key: K,
    value: V,
}
```
All three fields sit in the same cache line, minimizing memory loads during probing. This is a **non-obvious micro-optimization** — typical designs separate occupancy tracking.

**Robin Hood Backward-Shift Deletion** (lines 166–185):
```rust
fn rehash_from(&mut self, deleted: usize) {
    let m = self.mask();
    let mut gap = deleted;
    let mut idx = (deleted + 1) & m;
    while self.entries[idx].occupied {
        let ideal = self.hash_code(self.entries[idx].key) & m;
        let dist = (idx.wrapping_sub(ideal)) & m;
        let gap_dist = (gap.wrapping_sub(ideal)) & m;
        if dist > gap_dist {
            self.entries.swap(gap, idx);
            self.entries[idx].occupied = false;
            gap = idx;
        }
        idx = (idx + 1) & m;
    }
}
```
Uses `wrapping_sub` for boundary wrap-around — production-quality, not a textbook implementation.

**Float Hashing** (`hashable_float.rs`, lines 35–54):
- Converts f32/f64 to bit-level integers via `.to_bits()`
- NaN handling is correct: `to_bits()` produces consistent bit patterns
- Test at line 594: `m.insert(f32::NAN, 99); assert_eq!(m.get(f32::NAN), Some(99));` proves correctness

**Memory Management**:
- Resizing: `(old.len() * 2).max(DEFAULT_CAPACITY)` — standard exponential growth
- Capacity always power-of-two — enables `& mask` for fast modulo
- Immutable collections: `Arc<[T]>` for O(1) sharing without per-element overhead

**Algorithm Choices**:
- TreeMaps/TreeSets delegate to `std::collections::BTreeMap`/`BTreeSet` — correct engineering decision (don't reimplement what stdlib does well)
- HashBag uses `HashMap<value, count>` internally — no duplicate storage
- Stream generators use `std::iter::successors()` and `std::iter::from_fn()` for lazy evaluation

## 4. Correctness

### Edge Cases Handled

- **Float NaN**: Bit-level comparison via `to_bits()` — NaN keys work correctly (tested)
- **Boolean keys**: Only 2 possible values, correctly converts to 0u64/1u64
- **Large-scale operations**: `map_delete_heavy` test — 50,000 inserts, 25,000 removes, 25,000 more inserts, remove all 75,000. Verifies Robin Hood deletion under churn.
- **Empty collections**: Every operation checks `len() == 0` or `is_empty()` before iteration

### BiMap Synchronization Logic (`i32_i64_hash_bi_map.rs`, lines 35–47)

```rust
if let Some(old_key) = self.reverse.insert(value, key) {
    if !(old_key == key) {
        self.forward.remove(old_key);
    }
}
```
Elegant handling of the case where `(K1 -> V)` becomes `(K2 -> V)` — avoids unnecessary deletion when same key is re-inserted.

### Panic Safety

- `.set()` and `.remove_at_index()` on ArrayList documented with `# Panics` section
- These are intentional API choices matching Java's `List.set()` semantics

## 6. Evidence of Human Craftsmanship

### Non-Obvious Design Choices

1. **Interleaved Entry Layout**: Not auto-generated; hand-designed base abstraction. Code comment explicitly explains cache locality rationale. Subsequent specializations reuse it — upfront architectural investment.

2. **Robin Hood Hashing with Wrapping Arithmetic**: `wrapping_sub` for boundary cases. Not a standard teaching implementation — suggests study of academic literature (Sebastiano Vigna's work or similar).

3. **Float Bit-Representation Hashing**: Not the obvious approach (f32/f64 aren't `Hash` in Rust). Developer considered multiple solutions and chose simplicity.

4. **BiMap Consistency Logic**: The `if !(old_key == key)` guard before removing — shows thought given to edge cases that are easy to miss.

5. **Immutable Collections with `Arc<[T]>`**: Choice of `Arc<[T]>` over `Rc<Vec<T>>` reflects understanding of atomic reference counting, slice patterns, and thread-safety.

### Iterative Refinement Signs

- **Test coverage progression**: Basic tests in all types, but advanced tests like `map_delete_heavy` (75,000-item scenarios) suggest early bugs in deletion led to scale tests
- **Comment density**: Core implementations (hash_table.rs) have detailed comments; generated code has minimal comments (as intended)
- **Stream generators**: Uses `std::iter::successors()` and `std::iter::from_fn()` — leverages stdlib rather than rolling custom iterators

## 7. Areas for Improvement

### Minor Issues

1. **ArrayList.distinct()** uses `HashSet::insert()` return value as filter predicate — works but unconventional. Could be clearer.

2. **Limited stream terminal operations**: Missing `reduce`, `min_by_key`, `max_by_key`. Acceptable since Rust iterators provide these via `.fold()`, `.min()`, `.max()`.

3. **Documentation**: Lacks performance characteristic notes (Big-O, memory overhead, cache behavior).

4. **No generic immutable Multimap**: Would require `Arc<HashMap<K, Arc<Vec<V>>>>` — reasonable but not generated.
